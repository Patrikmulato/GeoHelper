# Copilot Instructions for GeoGuessr Helper

## Required Node Version

**Node.js 22 or higher is required for this project.**

This is enforced by:

- `package.json` `engines` field
- `.nvmrc` (for nvm users)
- `.node-version` (for fnm/other tools)
- CI workflow uses Node 22

If you're getting Husky errors or module resolution issues, ensure you're on Node 22+. The project uses Husky v9 which requires Node 20+, and some dependencies have better support on Node 22+.

## Documentation Sync Rule

**IMPORTANT:** When making changes to any of the following files, ALWAYS update the corresponding documentation files:

### Configuration Files That Trigger Documentation Updates

- `package.json` — npm scripts, dependencies, lint-staged config
  - Use exact versions only; do not use caret ranges like `^` in dependencies or devDependencies.
- `.husky/*` — Git hooks
- `eslint.config.mjs` — Linting rules
- `jest.config.js` — Test configuration
- `tsconfig.json`, `tsconfig.build.json` — TypeScript settings
- `next.config.ts` — Next.js configuration
- `.github/workflows/*.yml` — CI/CD workflows
- `.prettierrc.js`, `commitlint.config.js` — Code formatting and commit rules

### Documentation Files to Update

When changes affect the above config files, update these documentation files:

1. **`.github/copilot-instructions.md`**
   - Commands section
   - Pre-Commit Hooks section
   - Any configuration-related sections

2. **`AGENTS.md`**
   - Commands table
   - Git Hooks section
   - CI/CD section
   - Any convention that changed

3. **`CLAUDE.md`**
   - Quick Reference section
   - Any command shortcuts that changed

### Update Rule Examples

- **If you add a new npm script** → Update `## Commands` sections in all three files
- **If you modify ESLint config** → Update linting sections
- **If you add/change a Husky hook** → Update the `## Git Hooks` sections
- **If you modify CI workflow** → Update the `## CI/CD` section

### How to Update

1. Review the change made
2. Identify which documentation files are affected
3. Update the affected sections with accurate information
4. Ensure examples match the actual configuration
5. Keep formatting consistent (tables for commands, bullet lists for hooks)

### When in Doubt

Ask yourself: "Would a developer using this repo need to know about this change?" If yes, document it.
